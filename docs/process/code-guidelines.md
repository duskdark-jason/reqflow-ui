# 代码规范

本文件用于沉淀项目实现细则。`AGENTS.md` 只保留入口、红线和验证要求；具体语言、框架、组件、数据库、接口、日志和样式规则写在这里。

## 使用方式

- 按项目技术栈改写本文件，不保留无关框架规则。
- 规则要基于当前项目事实，不写“可能使用”的泛化判断。
- 高风险规则应写清触发场景、必须检查的文件和验证方式。
- 如果规则只适用于某个业务模块，应优先沉淀到 `docs/ai-harness/modules/` 或 `docs/domains/`。

## 建议章节

- 语言和框架规则。
- 数据访问或 API 调用规则。
- 组件、页面或样式规则。
- 权限、安全和日志规则。
- 常见错误和必须复用的本地工具。

## 代码注释规则

代码变更必须判断是否需要补充注释，并在 `execution-report.md` 的“代码注释处理”中记录结果。

必须写注释的场景：

- 复杂业务分支、状态流转、权限边界、兼容逻辑或失败降级逻辑。
- 数据库 join、统计口径、分页粒度、数据去重、缓存一致性或历史数据兼容。
- MCP 协议、外部系统对接、模板替换、安全边界或跨仓协作约束。
- 为绕开框架限制、历史包袱、环境差异或第三方行为而做的非直观处理。
- 修改已有复杂逻辑时，如果原逻辑没有注释，必须补一条能解释业务原因或风险边界的注释。

注释写法：

- 优先使用中文说明业务原因、约束、不变量和风险，不解释语法本身。
- 注释放在复杂代码块、非直观分支或关键方法前；避免每行流水账式注释。
- 不写“设置变量”“调用方法”这类重复代码表面的注释。
- 如果确实无需新增注释，必须在执行报告中说明“逻辑直接、无复杂业务分支/外部约束，因此未新增代码注释”。
