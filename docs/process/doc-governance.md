# 文档治理规范

## 文档分类

| 类型 | 推荐位置 | 说明 |
|---|---|---|
| Agent 执行护栏 | `docs/ai-harness/` | 模型和同事修改代码前读取 |
| 业务领域入口 | `docs/domains/` | 当前业务边界、领域索引和长期入口 |
| 新需求说明 | `docs/specs/active/` | 开发中的需求 |
| 已完成需求 | `docs/specs/done/` | 完成后的临时归档区；发布前可清空，仅保留 `.gitkeep` |
| 流程规范 | `docs/process/` | 新需求、文档维护、验收流程 |
| 模板 | `docs/templates/` | 可复制的新文档模板 |
| 运行手册 | `docs/runbooks/` | 本地启动、部署、联调、排查 |
| 历史资料 | `docs/archive/` | 旧计划、过期草稿、一次性资料 |

## 更新触发条件

- 新增模块、接口、页面、导出或任务。
- 修改接口字段、状态枚举、错误语义或权限行为。
- 修改数据库表、字段、join、统计口径或分页粒度。
- 修改用户可见核心流程。
- 修改构建、部署、联调或验收方式。
- 新增、拆分或重命名模块文档、契约文档、决策文档、搜索关键词或长期入口。

## 编写要求

- 一个文档只承担一个职责。
- 各类落地文档必须使用中文描述；必要英文术语、命令、接口名和工具名可以保留，但标题和说明必须给出中文语义。
- 新需求 spec 目录必须使用 `REQ-001-中文需求标题` 形式；中文目录用于团队阅读，Git 任务分支使用 ASCII。
- 单个业务 harness 文档尽量控制在 300 行以内；超过后优先拆分为模块、契约或决策文档，并同步更新 `docs/ai-harness/search-map.md`。
- 长期有效规则写清“不变量”和“验证方式”。
- 搜索导航必须写清关键词、功能/场景、入口文档和代码入口，避免让模型从全仓搜索开始理解项目。
- 模板中的占位符使用 `【】` 包裹，复制后必须替换。
- 文档或 harness 变更后运行 `sh scripts/check-docs.sh`。
