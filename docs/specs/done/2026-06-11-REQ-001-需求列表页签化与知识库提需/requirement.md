# 需求列表页签化与知识库提需需求说明

## 背景

需求人员在“需求列表”新增或修改需求时，当前表单仍在列表页弹窗中完成，模块下拉依赖人工模块数据，不能稳定读取项目分支知识库中的页面业务功能。影响范围字段也暴露给提需求人，增加了录入负担。

## 目标

- 新增和修改需求改为打开隐藏页签，列表页只承担查询、状态流转和入口跳转。
- 需求表单按项目和项目分支读取知识库模块，同时允许录入一个尚未存在的新功能需求名称。
- 影响范围不再作为表单区块展示，前端在保存前按知识库推荐自动回填后端字段。
- 新增需求时不展示需求编号，后端生成后再在列表和详情展示。

## 范围

本次包含：

- 新增需求维护页签路由和页面，复用现有列表表单逻辑。
- 调整需求列表的新增、修改入口，使其跳转到维护页签。
- 需求维护页签从 `/requirement/index/module/tree` 读取知识库模块，并支持新功能名称输入。
- 隐藏表单影响范围字段，保存时自动调用影响面推荐并填充提交字段。
- 同步前端契约和模块 harness 文档。

本次不包含：

- 不新增左侧菜单。
- 不改变需求状态流转。
- 不改变项目初始化和索引发布流程。
- 不调整持久化结构或脚本。

## 影响范围

- 接口/API：是，复用 `/requirement/index/module/tree` 和 `/requirement/index/impact/suggest`，新增需求仍调用 `/requirement/demand`。
- 持久化结构/脚本：否。
- 权限/菜单：否，新增隐藏路由复用 `req:demand:add`、`req:demand:edit`、`req:demand:query`。
- 页面/交互：是，影响需求列表、新增维护页签、修改维护页签和需求详情模块展示。
- 导出/异步/任务：否。

## 契约与数据口径

- 模块知识库一行对应一个项目分支下的索引模块或功能点。
- 需求表单选择已有模块时，提交该模块对应的兼容 `moduleId` 或索引模块标识；录入新功能时，允许不提交已有模块标识，并通过需求备注兼容保存新功能名称。
- 影响范围字段由系统根据 `projectId + variantId + moduleCode` 自动关联推荐；没有推荐内容时提交为空，不阻断需求保存。
- 新增需求不接受用户输入的需求编号；编号由后端保存时生成。

## 验收标准

- AC-UI-001：在需求列表点击新增或修改，会打开独立页签，列表页不再弹出需求编辑 dialog。
- AC-UI-002：新增需求页面不展示需求编号输入；修改页面可展示只读需求编号。
- AC-UI-003：选择项目和已初始化分支后，模块下拉读取知识库模块，并按 `projectId + variantId` 过滤。
- AC-UI-004：表单允许录入新功能名称，未选择既有模块时仍可提交功能需求。
- AC-UI-005：影响范围区块不在表单中展示；保存前自动按知识库推荐填充影响字段。
- AC-UI-006：需求详情和列表能正确展示知识库模块或新功能名称。
- AC-UI-007：前端构建和文档/harness 检查通过。
